## 为什么引入循环一致性损失 (Cycle Consistency Loss)？

引入循环一致性损失是为了解决如何在没有配对样本的情况下训练生成器的问题。在没有循环一致性损失之前，通过Gan训练生成器是用的Pix2Pix的方式，这种方式是需要配对样本并结合L1损失来训练。比如我想训练生成器把正常颜色的苹果转成灰色，那么我就需要正常颜色的苹果和对应的灰色图。生成器把正常颜色的苹果生成灰色，再通过L1损失不断优化生成器。判别器来判断是否是真实的图片，如果不是就继续优化生成器。最终训练一个判别器无法判断真假的生成器。

但是真实世界中很难获得这种配对样本，所以我更容易获得一堆正常颜色的苹果，和灰色图苹果。循环一致性损失就能够在这种没有配对样本的情况下通过一堆正常颜色苹果和一堆灰色图苹果来训练一个网络，这个网络可以把正常颜色的照片转换成灰色图的照片。具体流程是使用两对GAN网络和两个循环。前向循环利用两个GAN网络，把X域的x迁移到Y域y。再把Y域y迁移回到X域x'。计算x和x'的损失。后向循环把Y域中的y迁移到X域中的x，把X域中的x迁移回Y域中的y‘，计算y和y’的损失。最后通过L2损失结合前向循环的损失和后向循环的损失优化这两个生成器和判别器。

## 详细描述一下你使用的生成器和判别器 的架构。为什么生成器中要使用残差块？

生成器通过编码器对原始图片进行编码操作，之后使用resnet对编码的结果进行风格迁移，最后再通过解码器恢复图片的大小，达到风格迁移。判别器是将输入的图片进行卷积操作，得到30*30的特征图，每个像素点对应原先图片70*70的像素。特诊图矩阵上面的每个值对应原先图片的真实度。对这个真实度进行求均值方差得到最后的计算损失，看是真实图片还是假图片。生成器采用残差块是因为，风格迁移只针对部分部门区域进行风格迁移，残差块能够方便的操作哪些地方需要迁移，哪些不需要。

## CycleGAN 的总损失函数由哪几部分组成？其中 Identity Loss (恒等映射损失) 有什么作用？如果去掉它会有什么后果？

损失包含对抗损失、循环一致性损失、恒等映射损失。对抗损失的作用就是看马生成的斑马是不是真的像斑马。循环一致性损失就是让生成的斑马不能乱生成，要保留原来马的结构，要求变成的斑马映射成马的时候依然能够复原。恒等映射损失的作用就是对于马-斑马的模型，如果输入的是斑马的照片，就不要再转换了，否则颜色很能会变化。这三个损失中最总要的是循环一致性损失。如果去掉 Identity Loss，生成器可能会粗暴地对所有输入施加统一的颜色偏移），虽然不影响循环一致性的满足，但会导致输出的色调不可控

## 代码中使用了 Instance Normalization 还是 Batch Normalization？在图像生成任务中，为什么通常首选 Instance Norm？

使用的是Instance Normalization，因为 Batch Normalization在归一化的过程中会把不同图片的特征信息混合在一起，导致生成的每一张图片色调对比度都差不多。而Instance Normalization是独立计算归一化，不会丢失原先信息的色调、对比度等。

## 你并在简历中提到“代码重构与解耦”，具体是指哪些部分？官方源码中的 DataLoaer 或 Options 模块有什么问题，你是如何改进的？

官方源码是一个通用型的训练框架，同时支持 CycleGAN、Pix2Pix 等多种模型。因为训练和推理共用一套代码体系，即使只做推理也必须走完整套流程，需要填写推理中许多用不到的参数。针对官方文件耦合性较高的问题，我做了一下三点改进。

首先，在模型加载方面，官方的文件如果需要加载生成器、他会把判别器、损失函数和优化器所使用的库全部加载上去。但实际上推理的时候只需要用到生成器进行前向传播，因此我把加载生成器这一部分单独提取了出来。

第二，在数据读取方面，官方的文件需要通过多线程从文件夹中批量读取图片到内存，但在我的场景中用户通过Gradio直接把图片传入到了内存里，我省去了读取图片的这一步骤。

第三，在参数配置方面，官方的推理是通过命令行脚本启动的，指定数据集路径等信息。但我的场景是 Web 应用，把推理需要的固定参数直接写在代码里，让推理变成一个可以随时调用的 Python 对象，前端传入参数之后直接可以启动推理。

最终效果是从原来 20 多个文件简化成 1 个文件，前端选择模型类型、上传图片，后端就能直接返回结果。

## 你是如何实现 GPU/CPU 自适应切换的？在没有显卡的机器上，你的代码能正常运行吗？

我在代码初始化的时候做了一个自动检测，程序启动时先判断当前机器有没有 GPU，有就用 GPU，没有就自动切到 CPU。然后在整个推理流程中，模型、权重、输入数据全部统一放在检测到的那个设备上，保证它们始终在同一个设备运行，这样就不会出错。

## 请描述一下从用户上传图片到显示结果的完整数据流。Gradio 接收的图片格式是什么？送入模型前做了哪些预处理

用户上传一张图片后，Gradio 会把它转成 Python 里标准的图片对象传给后端。后端拿到图片后做了三步预处理：第一步是统一尺寸，不管原图多大都缩放到 256×256，因为模型训练时用的就是这个尺寸；第二步是转成张量，把图片从普通的像素数据变成 PyTorch 能计算的张量格式；第三步是归一化，把像素值从 0 到 255 的范围压缩到 -1 到 1 之间，匹配模型训练时的数据分布。预处理完就送进生成器做前向推理，模型输出的也是一个 -1 到 1 的张量。然后我做了反向的后处理：把值域从 -1 到 1 映射回 0 到 255，再从张量转回普通的图片数组，交给 Gradio 显示在前端。

## 在 Gradio 开发中，你是如何处理 PyTorch Tensor 到前端可显示图像的转换的？

第一个是维度顺序不同。PyTorch 里 Tensor 的排列方式是"通道在前"，也就是先说颜色、再说高度和宽度。但前端图片的格式是"通道在后"，先说高度、宽度、最后才是颜色。所以我做了一个维度转置，把顺序调过来。第二个是数值范围不同。模型输出的值在 -1 到 1 之间，但正常图片的像素值是 0 到 255。所以我做了一个线性映射，把 -1 到 1 等比例拉伸回 0 到 255。处理完这两步之后就是一个标准的图片数组了，Gradio 可以直接识别并展示给用户
