# 请用通俗的语言描述 RAG (检索增强生成) 的完整工作流程。
	RAG类似于外挂数据库，工作流程分为三步。第一步通过关键词匹配和向量语义检索两种方式，在知识库中搜索相关文档，再通过 RRF 融合排序算法，根据文档在两路检索中的排名计算综合得分，筛选出最相关的文档。第二步构建prompt提示词。把检索出来的排名靠前的文档和用户问题拼成提示词喂给大模型。第三步生成。大模型通过参考第二步构建的提示词生成最终的答案
# 为什么用关键词匹配和向量语义检索这两种方式？
	我使用 BM25 和向量检索这两种方式，是因为单一检索方式存在局限性。BM25 是基于关键词匹配的，它对精确的专业术语检索效果好，但无法理解语义；而向量检索是把文本转化为向量再计算相似度，可以理解同义词和近义词，但对精确的专业词汇可能不够精准。两者是互补关系，结合使用可以同时保证精确匹配和语义理解的能力，提高检索的召回率和准确率。
# 你是怎么筛选出最相关的文档
	目前每路检索 10 个、最终保留 10 个，这是一个初始设置。更理想的做法是扩大初筛范围再收窄，比如每路召回 20~~30 个，融合后取 Top 5~~10 个，这样能让 RRF 的排序更充分。具体的参数需要通过消融实验来确定最优值，这是后续可以改进的方向。
# 简述RRF算法
	RRF 是一种多路检索结果的融合排序算法。对每个文档，计算它在各路检索结果中排名的倒数之和作为最终得分，然后按总得分从高到低排序。这样做的好处是可以筛选出最相关的文档。（参数m，1/60,1/61.m越大区分度越小不会因为一路排名特别靠前就大幅拉高总分。m越小排名靠前的文档得分优势越大，越"激进。m=60是论文得出的最合适的一个值。k越小，如k=5，噪声越小、更精炼、但可能遗漏有用信息；k越大，如k=20，保留更多文档，信息更全，但 Prompt 更长，可能引入噪声，且消耗更多 token。）
	
# 为什么选择 Chroma？它和 FAISS 或 Milvus 相比有什么特点？`persist_directory` 的作用是什么？
	Chroma 部署简单、内置持久化、与 LangChain 集成方便，对于我这个万级数据量的项目完全够用。其他数据库比如FAISS或Milvus没有内置持久化或者部署困难，使用起来不方便。

# chunk是怎么切的？
	我使用的是langchain按的固定长度加分隔符的方式，因为我的医疗数据本身就是一问一答的格式，每条问答独立成行，并且大部分问答对的长度都在500字符以内。所以我以500字符为最大chunk，以换行符作为分隔符，这样切分之后几乎每个问答对都是一个chunk。另外由于每条问答之间没有上下文依赖关系，所以不需要设置chunk重叠。

# 你的embedding用的什么模型，为什么用这个。你还了解其他向量模型吗，为什么不用其他的模型，比如字节的或者腾讯的
	我使用的是阿里云 DashScope 的 text-embedding-v4 模型。这是阿里专门针对中文优化的向量模型，对中文医疗术语的语义理解比通用模型更准确。我早期使用过智源的BGE模型，这是开源模型可以下载到本地，但因为我推理用的是通义千问的模型，为了保持工程上的统一性，把BGE换成了通义千问的模型。

# 项目中使用了 `bitsandbytes` 的 4-bit 量化 (NF4)。请问量化主要解决了什么限制？对模型精度有多大影响？
	4-bit 量化主要解决的是**显存限制**。Qwen2-7B 原始精度需要约 28GB 显存，我的显卡只有 16GB，无法直接加载。通过 NF4 量化，将显存需求压缩到约 3.5GB，使得模型可以在消费级显卡上运行。精度方面会有轻微损失，但在我的医疗问答场景中影响很小，因为模型主要是基于 RAG 检索到的参考资料做信息整理，不需要强推理能力。
